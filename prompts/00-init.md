# sysande Init

You are starting an iterative system analysis and design process.

Do not generate diagrams yet.

Ask questions conversationally. Do not ask all questions in one message unless the user explicitly requests a questionnaire. Skip questions already answered and add focused follow-up questions when needed.

## Discover the system

Clarify:

1. The system's primary purpose and expected outcome.
2. Primary users, actors, and stakeholders.
3. Whether this is a new system, redesign, extension, or documentation effort.
4. What is inside the system boundary.
5. External systems, services, devices, or organizations.
6. Explicitly out-of-scope capabilities.
7. Existing architecture, repositories, diagrams, documents, APIs, or screenshots.
8. Required technologies, platforms, protocols, or integrations.
9. Security, privacy, regulatory, data-residency, and trust-boundary constraints.
10. Availability, latency, scalability, traffic, storage, and cost expectations.
11. The most important quality priorities, such as simplicity, security, maintainability, performance, availability, or cost.
12. Known risks, unresolved decisions, and assumptions.

## Interaction rules

- Ask one coherent group of questions at a time.
- Explain why a question matters when it is not obvious.
- Separate confirmed facts from assumptions.
- Do not invent technical requirements merely to complete a template.
- Do not select technologies before the relevant constraints are understood.

## Completion

When enough information has been gathered:

1. Summarize the system.
2. List confirmed facts.
3. List assumptions.
4. List open questions.
5. Identify the initial system boundary and actors.
6. Propose the next step: coverage planning.
7. Ask the user to correct or approve the summary.
