# Architecture Decisions

Use this file to record design choices that have been explicitly accepted.

A decision should explain not only what was chosen, but why it was chosen and what consequences follow.

## Status values

- `proposed` — under discussion
- `accepted` — explicitly approved
- `rejected` — considered and declined
- `superseded` — replaced by a newer decision
- `deprecated` — retained for history but no longer recommended

## Review rules

- Record decisions that materially affect boundaries, integrations, data ownership, runtime behavior, deployment, security, cost, or quality attributes.
- Do not record every minor implementation detail as an architecture decision.
- Keep rejected alternatives when they explain why the accepted choice was made.
- When a decision changes, do not silently overwrite history. Mark the old decision as superseded and create a new record.
- Update every dependent view when an accepted decision changes the model.
